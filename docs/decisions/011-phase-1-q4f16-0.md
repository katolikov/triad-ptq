# ADR-011 — Phase 1: q4f16_0 layout swap as opt-in MLC export option

Status: **proposed (host-side change committed; on-device measurement deferred)**
Date: 2026-05-05
Author: Claude (autonomous engineer)

## Context

The MLC weight-pack format `q4f16_1` (interleaved (scale, zero) per-group,
INT4 codes packed into uint32 lanes) is the canonical Android default and
is what the prior session shipped (STATUS.md). The
[callstack.com](https://callstack.com) Adreno write-up on Llama-3.2-3B and the mlc-llm
"Known Issues" doc both report that `q4f16_0` (struct-of-arrays —
weights and per-group params in separate tensors, no interleave) is
~5–10 % faster on Adreno-class mobile GPUs because compute shaders can
issue contiguous LDS / `local` loads aligned to wave reads. The AMD RDNA
Performance Guide §"To help maximize bandwidth in compute shaders, write
to images in coalesced 256-byte blocks per wave" is the matching
guidance on the AMD side.

Phase 1 of this session asks us to test whether that observation
generalises to the Xclipse 950 (RDNA-mobile heritage).

## Decision

`experiments/14_export_mlc.py` now accepts `--quantization
{q4f16_0, q4f16_1, both}` (default `q4f16_1`, preserving prior shipped
behaviour). The export pipeline reuses the same TRIAD-folded HF
safetensors and runs `mlc_llm gen_config + convert_weight + compile`
once per requested quantization, emitting per-format bundle dirs:

* `q4f16_1` only (default) → `/tmp/triad-tinyllama-int4-mlc/`
* `q4f16_0` only           → `/tmp/triad-tinyllama-int4-mlc/`
* `both`                   → `/tmp/triad-tinyllama-int4-mlc-q4f16_{0,1}/`

The summary JSON now contains a `bundles` array so multiple bundles are
captured side-by-side for the device bench.

## Acceptance gate (from session prompt)

* PPL parity (PPL(_0) − PPL(_1) ≤ 0.005). The two layouts are
  numerically equivalent up to fp16 rounding — both run the same matmul
  kernel against dequantised weights — so PPL parity is virtually
  guaranteed and is verified by the existing perplexity evaluator on
  the host (CPU-fp32 dequant path), not the device. The host PPL
  invariance test belongs in `tests/test_mlc_export_layout.py` once
  the bundles exist (filed as TODO at end of this ADR).
* `tok/s_decode(_0) − tok/s_decode(_1) ≥ +1.5` **OR** `prefill ≥ +2`.
  Requires a working device runner (see ADR-009) plus a built APK
  bundle for each quantization. Neither is in this session's reach.

## Why this ships now without device numbers

The export option itself is host-side and zero-risk: it routes through
`mlc_llm` exactly as the existing `q4f16_1` path does, with a different
flag. Committing this code now means:

1. The TRIAD calibration / safetensors export work need not be redone
   when a future session has the device runner ready — only the final
   `mlc_llm compile` step has to run for the new layout.
2. The session's later phases (3, 4, 5) can chain on this branch and
   reuse the pipeline.
